Sélectionner les algorithmes adéquats pour répondre à un problème donné

AC 1 – Structures de données complexes

Dans mon stage chez Izydesk, j'ai utilisé un context global combiné à un reducer pour centraliser les données de l'entreprise. Cette structure permet d'éviter les redondances d'appels API et facilite la gestion d'état sur l'ensemble du site. Elle est bien plus adaptée qu'un simple useState pour un projet e-commerce à composants multiples.

AC 2 – Techniques algorithmiques adaptées

Sur le projet Vote Électronique, j'ai implémenté l'algorithme de chiffrement ElGamal en Java. Ce choix permettait de garantir la sécurité des bulletins tout en respectant les contraintes d'un système client-serveur. J'ai également utilisé des structures mathématiques adaptées aux grands entiers.

AC 3 – Sécurisation des données et du code

Dans le site e-commerce PHP, j'ai sécurisé les formulaires contre les injections SQL à l'aide de requêtes préparées. J'ai aussi mis en place une gestion des sessions utilisateurs pour protéger les accès aux commandes.

AC 4 – Impact environnemental et sociétal

Lors de la Nuit de l'Info, nous avons conçu un jeu web léger avec JavaScript et Canvas API, pensé pour être accessible sur tout type d'appareil sans surconsommation de ressources. Nous avons privilégié un code simple, peu gourmand, afin de réduire l'impact énergétique du site sur le long terme.

AC 1 – Optimiser les modèles de données

Dans le projet E-commerce PHP, j'ai conçu un modèle relationnel normalisé pour la gestion des utilisateurs, produits et commandes. J'ai adapté les relations pour permettre des jointures efficaces dans les requêtes critiques (panier, historique, etc.).

AC 2 – Assurer la confidentialité des données

Lors de mon stage chez Izydesk, j'ai sécurisé les données utilisateurs en contrôlant systématiquement les erreurs API et en affichant des messages adaptés en cas d'anomalie. Le système de panier, par exemple, est protégé contre les manipulations côté client.

AC 3 – Organiser la restitution de données

Dans le site e-commerce PHP, j'ai développé des vues HTML dynamiques pour afficher les produits, les détails de commande et les informations de compte. Côté back-end, les données étaient extraites via des classes Model en PHP puis injectées dans les View.

AC 4 – Manipuler des données hétérogènes

Chez Izydesk, j'ai manipulé des objets JSON complexes retournés par l'API. J'ai dû filtrer dynamiquement des produits non affichables à l'aide de l'attribut isDisplayForPos, ce qui m'a obligé à adapter le rendu du front-end en fonction de la logique métier.

Appliquer une démarche de suivi de projet en fonction des besoins métiers des clients et des utilisateurs

AC 1 – Identifier les processus d'une organisation

Pendant mon stage chez Izydesk, j'ai analysé les flux de gestion de commande dans le secteur de la restauration, et intégré ces besoins métiers dans la conception du site (types de commande, options produits, allergènes, gestion d'adresses).

AC 2 – Formaliser les besoins du client

J'ai rédigé un cahier des charges personnel en début de stage chez Izydesk pour structurer mes objectifs. Ce document comprenait les besoins fonctionnels comme l'ajout au panier, la navigation par catégorie ou l'internationalisation.

AC 3 – Identifier les critères de faisabilité

Lors de la Nuit de l'Info, nous avons évalué dès le départ ce qui était réalisable en 24h. Nous avons réduit les ambitions techniques et nous sommes concentrés sur un MVP : un jeu simple, responsive et fonctionnel. Ce tri initial nous a permis de livrer dans les temps.

AC 4 – Définir une démarche de suivi de projet

En stage, j'ai suivi un cycle structuré avec GitLab : création de branches, merge requests, validation par le CTO. Chaque semaine, un point d'équipe permettait de suivre l'avancement et de prioriser les tâches. Ce cadre m'a permis de livrer régulièrement du code stable.
